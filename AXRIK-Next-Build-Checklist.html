C Build the habits that pre-solve the 5 rework causes

  • Ship role-based RLS in the first migration — admin / staff / account / public; financial tables admin-only from day one. Cause #1, the biggest win: security was hardened 4 times on JG Foods.
  • Use the corrected starter schema + triggers — slot-count updates on slot moves, signup trigger handles null metadata. Cause #2: edge cases were found in production.
  • Land the whole spine in 3 consolidated SQL files, not incremental migrations. Cause #3: 16 migrations grew one at a time.
  • Put config in an app_settings key/value table, never constants — delivery days, cut-offs, copy. Cause #4: every tweak was a code change.
  • Run the end-of-session hygiene sweep every time. Cause #5: stale placeholders shipped to the client.

D The prompts that drive it

1 · Kickoff (project start)New AXRIK client: [name], a [type] in [location]. Here are their order messages / spreadsheets / invoice format [attach]. Follow the kickoff workflow — propose the data model first (from the starter kit, tell me what to customise), then the build phases. Don't write screens until I've confirmed the schema.
2 · Stand up the backend (day one)Clone the starter-kit migrations for [client]. Customise the >>>-marked CHECK constraints: delivery days [days], order channels [channels], reference prefix [XX]. [Keep/drop] invoicing. Give me final 001/002/003 SQL ready to run, the run order, and the exact promote-owner-to-admin snippet with [owner email].
3 · Customer website (phase 1)Build the [client] website as a single responsive HTML file in their branding [colours/logo]. Lead with the business problem and outcomes, not tech. Sections: hero, about, how it works, product showcase live from Supabase, trade/wholesale path, contact form (EmailJS). Wire the order form to place_order RPC with the slot picker.
4 · Admin screen (phase 2, repeat)Add a [feature] page to the [client] admin app, following the existing showPage() shell. Render from a single live Supabase fetch on open, respect admin/staff RLS roles, read config from app_settings. Match existing pages. Add a matching entry to the in-admin HELP_PAGES guide.
5 · AI feature (phase 3)Add an AI [order-parse / message-redraft / social-post / summary] feature to [client], following the AI-features pattern. Route through the existing /.netlify/functions/ai proxy, build a non-AI fallback so the button always works without a key. AI drafts, client confirms before send/save.
6 · Build summary (end of session)Update build summary: [what was done this session]. (Replaces in place AND posts to the AXRIK portal via portal-update — same filenames, never numbered versions.)
7 · Go-live & handover (launch)[Client] is ready for go-live. Following the deliverables pattern, produce the go-live checklist, a client-runnable testing worksheet, and refresh the build summary. AXRIK black/white/green branding, first person as me, UK English.
8 · Client update email (any time)Draft a short update email to [client] about [what's new]. First person as me, warm, UK English, lead with the outcome, flag anything I need from them, end with what's next. No jargon.
9 · Hygiene sweep (every session)Before we finish: scan for leftover placeholders or stale integration notes, confirm every table has RLS, confirm every AI feature has a fallback, and list anything still stubbed so it doesn't surprise me next session.
